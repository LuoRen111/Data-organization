# 医疗Java开发实习面经(长沙自研小厂)

**面试时间**：2026\-04\-20
**投递岗位**：Java 开发实习岗（药品管理 / 药理系统方向）
**面试轮次**：综合面试（技术经理 沟通 \+ 技术面）

## 一、技术经理沟通环节（技术面负责人来晚了，所以跟技术经理聊的比较久）

### Q1：医疗行业开发职业规划方向？（录音晚了点，下面都是我的反问和沟通交流）

A：中级、高级，然后到架构师，然后到产品经理。做医疗的话，肯定最终往架构师或者产品经理去走。因为这两个岗位就是前面要求对整个业务的全盘掌控会有要求。人到了一定年纪做开发也是扛不住的，基本上到了三十四五岁以后，做开发确实就没那么跟得上，要提前做好职业规划。现在 AI 的话可能还可以往后延一点，因为更多的是做决策，AI 会帮你去完成一些代码的编写，这个时候个人对业务的理解要求更高。

### Q2：AI 对医疗行业的替代性看法？（反问）

A：做医疗来说，AI 没有可取代性。因为医疗业务规范里面很复杂，AI 只能在医疗永远是一个辅助性的东西，都是人去做决策。目前让 AI 去写一套完整的东西还是不太现实，只能写一些模块化的，而且要给的需求指令很清楚，要求对业务很熟悉，这都是个人能力，AI 没办法获取到。每个医院的流程不一样，充斥着太多个性化的东西。

### Q3：公司业务与技术栈情况？（反问）

A：业务是在原有模块化的基础上做二开，对原有模块做技术翻新，考虑数据的自动和整合，还有新的应用交互。后续会把公司边缘性业务如数据平台、OA 办公系统转成 java 来做。目前实习岗是做药品这一块，药理系统、药品管理，跟 ERP 有点像但又不一样。技术栈：spring cloud、mysql、国产化数据库 B7SQL、前端 Vue3，前后端分离。

### Q4：岗位核心要求？（反问）

A：要求性格开朗一点，需要去现场跟医院科室做需求调研和沟通。开发人员自己去了解用户的痛点难点，前期调研会让开发人员先把需求文档写好。做开发第一个技术功能实现很重要，第二个要自己多去思考产品能不能用、好不好用，把自己作为用户去感受。

### Q5：公司 AI 应用与落地情况？（反问）

A：用过 GPT、DeepSeek、豆包、Cloud Code、Codex 这些大模型，用来写一些简单的页面、整理接口规范。医疗大模型训练数据不足，隐私数据偏多，医院需要硬件本地化部署。可以本地部署增强智能检索知识库，上传文档、药品数据，解决隐私性和大模型幻觉问题。现在医院有语音病历、AI 病例、AI 辅助诊疗在使用，核心问题是隐私性、准确性、谁担责。

### Q6：公司团队规模？（反问）

A：后端开发将近 12 个，前端 3 个，架构师 1 个，产品 1 个，美工 UI1 个，测试 1 个；实施人员大概 4 个，项目多的时候临时招聘，有项目经理负责培训。

### Q7：医疗系统高可用保障？（反问）

A：这个可以留着等技术人员详细解答，医疗对数据安全还有稳定性要求非常高。

### Q8：加班与行业情况？(沟通交流)

A：emp定制化二开比较舒服，客户是国企、律所基本不需要加班。之前公司加班比较严重，有加班文化，没加班费也不让调休，实习生离职率高。

### Q9：当前就业与面试趋势？(沟通交流)

A：现在就业很卷，面试大部分看实操能力、场景题。企业不想花成本培养，想要直接能用的人。沿海裁员的技术人员回内陆，加剧内卷。

## 二、技术面试环节

### Q1：自我介绍

A：我叫\*\*，现在是现居在长沙，是\*\*\*\*大学信息管理信息系统的一个本科应届生。然后我具有一个良好的 java 后端开发技术和企业级的一个项目实战经验。在实战上的话，我是熟悉掌握了一个 java 核心，spring boot I mysql、redis\. 然后我还熟悉 JVM、GC 以及索引优化缓存三大问题的解决方法，能够独立的使用 JWT、kafka、ES、socket等组件进行后端开发。然后在实习期期间的话，我是在东华锐奇参与了一个瑞吉企业知识库的智能知识库的研发。负责大文件上传以及混合检索等核心模块的开发。通过 redis 优化、卡夫卡解耦、ES 向量化手段等，显著的提高了一个上传的上传和处理的效率。同时我还做过校园活动管理平台，能够解决高并发分布式锁缓存以及秒杀的一些问题。另外的话我还有竞赛国奖以及省奖，还有两项软著，学习能力比较强，能够快速上手 Java 后端业务开发，很高，很希望能够加入贵公司。

### Q2：秒杀接口开发注意事项？

A：这方面的话我觉得就是要从前端开始。前端的话就是很常见的一个地方，就是给那个按钮指挥，就点一下之后不准点第二下了。然后还有就是提前换提前还有提前缓存，就是可以放入本地缓存或者 redis 当中去做热点缓存。然后还有就是可以使用不同过滤器去做白名单校验，或者说是黑名单校验，这样子也可以做一层缓存。还有的话就是用那个令牌桶去做去控制单位时间内的请求量。然后就这些，后面的话还可以用卡夫卡去去做一些削峰解耦，还有一些其他的一些操作。

### Q3：微服务用过吗？

A：微服务的话用的比较少。实际场景的话还是没怎么使用过。我知道一些普通的一些应用，就比如说可以用 nacos 去做服务注册和发现，然后用 open feign 去做服务间的一个远程调用。还有我之前是用那个 get way 去做网关路由器，还有就是用 MQ 去做服务的一个异步解耦削峰，还可以保证一个消息的可靠，我大概就知道这些内容。实际都是那些正式工，或者说是公司一个架构是搭好的，然后我去用就好了。上家实习的时候是有参与过里面，是用过。

### Q4：能讲一下对于微服务跟单体服务的一个理解吗？

A：这个的话就是涉及到一个单体和单体服和微服的区别了。单体的话就是维护的优势就是有主要有几个优势，一个是能够扛住高并发，也不能这么说，就是他服务独立，因为他每就可以分成很多组件，然后可以放在很多不同的一个服务器上面。他的一个服务挂了，不会影响整个的一个系统。然后还有就是可以独立的一个可以去独立扩容。遇到一些像 redis 或者说 mysql，如果说这些压力比较大的时候，我们可以去独立的去增加一些机器来减少它的压力。然后就是开发的话可能更快。因为微服务的话可能可以把业务拆分，然后让每个团队去各自完成各自的任务，到时候可以一起合并。还有一个就是恢复的它的技术上更加灵活，就是每个服务都可以使用自己的一个语言或者使用自己的技术，不会被拘束。

### Q5：贵公司目前有对 AI 有一些应用或者说落地吗？（反问）

A：有，我们基于医生写那个电子病历，有基于这个 AI 制作，比如说做一些比如说一些病例的识别，比如说就比如说患者去去讲去描述他的症状，然后我们的 AI 可以去识别他的一些里面的关键信息，然后去把它汇总整理，整理成我们我们医生他需要的一些病例诊断的一病例的一些信息。我们目前是有做这一块。

### Q6：我们怎么去保证那个医疗系统的一个高可用？（反问）

A：分几个方面。第一个是比如说数据库，数据库的话我们会有像医院的话，它会有售后的集成，有主备库这种。然后城市的话基于像我们这边会做集成，然后我们各个服我们也不是只是一个单服，我们可能是多个服务器共同一起去去做搭建的一个集成，就会不就保证它不会有出现这种单点故障这些。
